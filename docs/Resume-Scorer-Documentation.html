Section 1

Executive Summary

Resume Scorer is a local-first recruitment assistant that turns a recruiter's email inbox into a scored, searchable, and actionable candidate pipeline. It pairs a Chrome (Manifest V3) extension with a small Node.js / Express backend so that resumes flowing into Outlook or Gmail are automatically extracted, scored by a large language model, indexed for semantic search, and pushed through configurable hiring automations — all without sending an API key to the browser.

The problem it solves

Recruiters receive resumes as email attachments and triage them by hand: open the PDF, skim it, guess a fit, copy details into a sheet, then chase scheduling over more email. Resume Scorer collapses that loop. The moment a resume lands, it is parsed, scored 0–100 with a category breakdown, enriched with structured candidate fields, made searchable in plain English, and — where the recruiter opts in — routed into automations that send assessment links, book interviews with live Google Meet links, or send a polite rejection.

What makes it different

🔒

Key-safe by design

Every AI call is brokered by the local backend. The extension never holds a provider key, and OAuth uses PKCE with no client secret.

💸

Cheap to run

Local embeddings (Transformers.js), score caching, and a regex-first query parser keep token spend near zero on the free tier.

🔎

Semantic search + RAG chat

"Show me backend devs with 3+ years" is answered by a SQL filter and a vector index, then narrated by the LLM.

⚙️

Visual automation

An N8N-style canvas wires triggers → filters → actions: send OA, schedule interviews, reject — with a full dry-run mode.

Project Scope

This section draws the boundary of the MVP: what the system does, what it deliberately does not, and the assumptions the design rests on.

2.1 In scope

CapabilityDescription
Inbox connectionOAuth login to Microsoft Outlook (Graph API) and Google Gmail, with automatic token refresh.
Attachment discoveryList recent emails that carry .pdf / .docx attachments; ignore inline images & signatures.
Text extractionServer-side parsing of PDF (pdf-parse) and DOCX (mammoth) into raw text.
AI scoring0–100 overall score plus a 5-axis breakdown (experience, skills, education, clarity, impact), summary, strengths, concerns, recommendations.
Candidate enrichmentStructured extraction of name, contact, current role/company, years of experience, education, skills, locations, remote history, certifications & more.
ClassificationAuto-categorisation into role categories and a role title for each candidate.
Semantic search & RAGLocal embeddings indexed in sqlite-vec (and optionally Pinecone) with FTS5 keyword search for hybrid retrieval.
AI chatCross-resume mode ("find me…") and single-candidate mode ("tell me about this person"), with a regex pre-parser to save tokens.
Recruiter dashboardWeb UI at localhost:8787 — KPIs, six charts, searchable candidate table, drag-drop upload, per-candidate modal.
Excel exportOne-click .xlsx export of candidates with scores & summaries (exceljs).
Automation builderVisual canvas: triggers, filters, branch/top-N logic, and actions (send OA, schedule interview, reject, notify, log) with dry-run.
Interview schedulingFinds the first free slot across selected interviewers' Google Calendars and books an event with a fresh Google Meet link.

2.2 Out of scope (for this MVP)

  • OCR of scanned / image-only PDFs — only text-bearing documents are parsed.
  • Multi-user accounts, roles & permissions — single local operator.
  • Cloud / multi-tenant hosting — backend runs on localhost.
  • Production hardening — rate limiting, audit logs, secrets vaulting beyond .env.
  • Deep ATS / HRIS integrations beyond Gmail / Calendar actions.
  • Scheduled / webhook triggers — the only automation trigger is "Manual run".
  • Mobile or non-Chromium browsers — targets Chrome / Edge (Chromium).
  • Bias auditing & compliance certification of the AI scoring model.

2.3 Assumptions & constraints

Operating assumptions
  • The operator runs the backend locally and keeps it running while using the extension.
  • The operator can complete a one-time Azure (Microsoft) and Google Cloud OAuth app registration.
  • An AI provider key (Groq is free) is available and placed in backend/.env.
  • Resumes arrive as machine-readable PDF/DOCX files in the connected inbox.
Hard constraints (architectural rules)
  • No API key may ever live in the extension — all model calls go through the backend.
  • Manifest V3 only — background logic is a service worker, not a persistent page.
  • OAuth is PKCE with a public client — no client secret is stored anywhere in the extension.
  • Text extraction & AI calls happen in the backend, never in the browser.
  • Secrets stay out of git — .env, token caches and node_modules are git-ignored.

3.3 Key scenario — Score a resume (UC-2)

Recruiter opens the extension & selects an email

The side-panel popup lists recent messages that carry PDF/DOCX attachments, fetched via Microsoft Graph / Gmail API using the stored access token.

Extension downloads the raw attachment

The attachment is fetched as base64 (contentBytes) — no parsing happens in the browser.

Extension POSTs to the local backend

POST /score { filename, contentType, contentBase64 } is sent to http://localhost:8787.

Backend extracts text & calls the LLM

extract.js decodes & parses the document; ai.js sends a strict-JSON prompt to the configured provider.

Backend persists, indexes & returns the result

The score, review & enriched fields are written to SQLite; chunks are embedded into the vector index; JSON is returned to the popup and rendered as a score + breakdown chart.

Edge cases the design handles
  • Duplicate scoring — a unique (email_id, attachment_id) index makes re-scoring the same attachment a no-op.
  • Scanned PDFs — if too little text is extracted, the UI fails loudly ("could not extract enough text").
  • Candidates with no email — automation marks them skipped, not failed.
  • Dry-run — every email & booking can be previewed (exact subject/body/attendees) before anything is sent.
  • Graph cycles — the automation engine refuses to run a workflow wired back on itself.

4.3 Data flow — RAG chat & semantic search

Query pre-parse

A regex layer extracts structured intent (category, years, skills, location, top-N) so deterministic filters run in SQL before any LLM call — saving tokens.

Hybrid retrieval

The query is embedded and matched against chunk_vectors (cosine) while FTS5 BM25 retrieves keyword hits; results are merged & ranked.

Context assembly

Top chunks plus structured candidate rows are packed into a compact grounding context.

LLM narration

The model writes the final natural-language answer (optionally streamed via /chat/stream), grounded only in retrieved evidence.

4.4 Data flow — automation run

Trigger

Operator presses Execute on a workflow (Manual-run trigger).

Filter & branch

Candidates flow through filters (score ≥ X, category, years, skills) and branch / top-N logic into action streams.

Actions

Send OA (Gmail), schedule interview (Calendar free/busy → event + Meet), polite rejection, notify, log — each paced to respect rate limits.

Run history

Every per-candidate result (sent / skipped / failed, with the API's error message) is recorded for audit. Dry-run renders everything without sending.

Section 5

Data Model

SQLite (via better-sqlite3) is the single source of truth, extended with sqlite-vec for vector search and FTS5 for keyword search. Pinecone and MongoDB are optional mirrors enabled by configuration.

5.1 Core tables

TablePurpose & notable columns
resumesOne row per scored candidate. Holds score, category, role_title, review_json, raw_text, plus enriched fields: candidate_name, email, phone, location, years_experience, current_title/company, top_skills, education_json, certifications, companies, domains, remote_worked, managed_people, and more. A unique partial index on (email_id, attachment_id) de-dupes re-scores.
chunksResume text split into retrieval-sized chunks; FK to resumes with cascade delete.
chunk_vectorsvec0 virtual table — fixed-dimension embeddings (EMBED_DIM; default 768) for cosine search.
chunks_ftsFTS5 virtual table mirroring chunk text via triggers — BM25 / keyword retrieval for hybrid search.
chat_threadsConversations; optionally bound to a single resume_id for per-candidate chat.
chat_messagesPer-thread message log (role + content), indexed by (thread_id, id).
automation tablesWorkflows (nodes/edges), run history, interviewers & availability windows, and message templates.

5.2 Storage strategy

StoreRoleBehaviour
SQLite + sqlite-vecPrimaryAlways on. Holds all relational data, vectors & FTS index in one local file (data.db).
PineconeOptional (scale)When PINECONE_API_KEY is set, vectors dual-write to Pinecone + sqlite-vec; reads prefer Pinecone and fall back to sqlite-vec on any error.
MongoDBOptional (mirror)Backfill / backup mirror of resume records for durability.
uploads/Blob storeOriginal PDF/DOCX bytes on disk; file_path references them for re-download / preview.
Graceful degradation
A Pinecone outage degrades to local sqlite-vec search rather than returning empty results — availability is never gated on an external vector store.

Section 8

Security & Privacy

🗝️

No keys in the browser

Provider keys live only in backend/.env. Extension code is fully readable, so it never holds a secret.

🪪

PKCE, no client secret

Auth-code + PKCE via a public client. SPA redirect URI enables CORS token exchange directly.

🏠

Local-first data

Resumes, scores & chats stay on the operator's machine in SQLite. No cloud account required.

🙈

Secrets out of git

.gitignore covers .env, token caches, node_modules & the database.

8.1 Token handling

  • Access / refresh tokens are stored in chrome.storage.local (extension) and refreshed ~60 s before expiry.
  • Backend-side Google OAuth tokens are obtained via the web flow and used only for the operator's own Calendar/Gmail actions.
  • Least-privilege scopes: Microsoft User.Read + Mail.Read; Google gmail.send, calendar.events, calendar.readonly.

8.2 Data minimisation & failure behaviour

  • Only attachments ending in .pdf / .docx are processed; inline images & signatures are skipped.
  • The UI fails loudly — login errors, empty inboxes and backend-down are surfaced, never silent.
  • Automation defaults to safety: dry-run preview, candidate-without-email → skipped, partial failures recorded with the upstream error.
MVP caveat
This is an MVP, not a hardened production system. Before any multi-user or internet-exposed deployment, add authentication on the backend, rate limiting, encrypted secret storage, and an AI-bias / fairness review of the scoring prompts.
Section 9

Setup & Deployment

Three one-time manual steps, then run two processes. Full step-by-step lives in the project README.md; this is the operational summary.

9.1 Prerequisites

  • Node.js 18+ (developed on Node 22) and a Chromium browser (Chrome / Edge).
  • An Azure app registration (Microsoft) and a Google Cloud OAuth client for automation.
  • An AI provider key — Groq is free (console.groq.com/keys).

9.2 Run the backend

#Action
1cd backend && npm install
2copy .env.example .env — fill GROQ_API_KEY (and Google values for automation).
3npm run dev → backend listens on http://localhost:8787.
4Open http://localhost:8787 for the recruiter dashboard.

9.3 Load the extension

#Action
1chrome://extensions → enable Developer mode → Load unpacked → select extension/.
2Open the service-worker console, copy the printed redirect URI (https://<id>.chromiumapp.org/).
3In Azure, add it as a Single-page application redirect URI; copy the Application (client) ID.
4Paste the client id into extension/config.js; reload the extension.
5Click the toolbar icon → Connect Outlook (and/or Gmail) → start scoring.
Definition of done
npm run dev starts the backend; loading extension/ unpacked shows a popup with "Connect Outlook"; after login the popup lists resume emails; clicking one renders a score + review; the dashboard shows the full pipeline.

9.4 Smoke tests

Automation logic is covered by node test-automation.js (29 tests — schema, CRUD, filters, template rendering, end-to-end dry-run, cycle detection, route handlers; no network calls).
